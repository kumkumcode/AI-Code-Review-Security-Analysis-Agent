# Milestone 1: Technical Data Schema & Mapping Study Guide

This document defines the structured information blocks utilized throughout our system. It describes how data is structured as it moves from the web portal through the AI processing pipeline, completely in plain English.

---

## 1. Schema A: Code Submission Package
*This represents the data structure package assembled the moment a developer uploads a file or paste codes.*

*   **Submission ID**: A unique, timestamped serial number generated automatically by the system to track the review session (e.g., `SUB-2026-0001`).
*   **Target Language**: The verified programming language of the file (specifically `python` or `java`). This tells the analyzer agents which structural syntax rules to execute.
*   **File Name**: The original user-provided filename (e.g., `auth_logic.java`). This helps pinpoint where bad code lives.
*   **Raw Code Source**: The complete string text of the submitted program file, ready for scanning.
*   **Timestamp**: The exact date and time the submission occurred, enabling chronological tracking.

---

## 2. Schema B: RAG Knowledge Base Document
*This represents the structured metadata stored alongside our vectorized Markdown files inside Chroma DB. This is what the Remediation Agent uses to retrieve clean-code lessons.*

*   **Document ID**: A clean, unique path-based identifier (e.g., `best_practices_java_resource_management`).
*   **Category**: The parent directory type of the rule (`owasp`, `secure_coding`, `code_smells`, or `best_practices`).
*   **Language**: The applicable language target (`python`, `java`, or `n/a` for general concepts).
*   **Source File**: The local relative path pointing back to the markdown document on disk.
*   **Content Body**: The raw markdown text containing the educational overview, bad examples, flawed attempts, and corrected code.

---

## 3. Schema C: Consolidated Agent Finding Alert
*This is the core schema generated by the analyzer agents. When a vulnerability or code smell is spotted, it is packaged into this structural model.*

*   **Finding ID**: A unique tracking key generated for the specific issue (e.g., `FIND-029`).
*   **Reporting Agent**: Specifies which agent raised the flag (`Security Agent` or `Code Analysis Agent`).
*   **CWE Reference**: The matching Common Weakness Enumeration ID (e.g., `CWE-89` for SQL Injection).
*   **Severity Score**: The threat grade assigned to the issue:
    *   `🔴 Critical`: Direct, highly exploitable security vulnerability (like unauthenticated password resets).
    *   `🟠 High`: Serious vulnerability or major structural issue (like SQL injections or severe resource leaks).
    *   `🟡 Medium`: Traditional security risk or code smell that reduces stability (like missing output validation).
    *   `🔵 Low/Info`: Best practice suggestions and minor clean-up recommendations.
*   **Location**: The exact line number in the source file where the issue begins.
*   **Short Description**: A brief, conversational explanation of the issue (e.g., *"Database connection is opened on Line 15 but is never closed."*).
*   **Remediation Details**:
    *   *The "Why"*: A plain-English explanation of why this is bad.
    *   *The Fix*: Step-by-step instructions on how to refactor the code safely.
    *   *Reference Code*: Side-by-side examples illustrating the insecure approach versus the secure solution.

---

## 4. Schema D: Global PR Summary Report
*This represents the final high-level metadata block compiled by the PR Summary Agent to generate the main page of the developer's report.*

*   **Report Metadata**: Includes the corresponding Submission ID, date of the review, and total analysis duration in seconds.
*   **Overall Grade Score**: An alphabetical safety and quality score assigned to the submission, ranging from `A` (perfectly safe) down to `F` (severe critical hazards).
*   **Alert Metrics**: Summary metrics compiling the aggregate count of flagged items:
    *   *Critical Count*: Total number of critical-severity security alerts.
    *   *High Count*: Total number of high-severity alerts.
    *   *Medium Count*: Total number of medium-severity alerts.
    *   *Low Count*: Total number of minor alerts.
*   **Roadmap Priority List**: A ranked sequence of instructions detailing exactly which files and line numbers the developer must modify first to repair the codebase's grade safely.
